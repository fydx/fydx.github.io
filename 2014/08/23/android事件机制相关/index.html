android事件机制相关

对于android的点击事件处理，一般会涉及到这三个方法

  • onTouch （存在于OnTouchListener中）
  • onTouchEvent （存在于View，ViewGroup，Activity中）
  • onInterceptTouchEvent （存在于ViewGroup中）

这三个方法的返回值都是Boolean

关于setOnClickListener的解读

平常用的最多的设置按键监听的方法是setOnClickListener，这是View的一个方法，在View中，我们可以找到这样一段代码：

public void setOnTouchListener(OnTouchListener l) {  
        mOnTouchListener = l;  
}

而我们在View的onTouchEvent中，我们又会看到这样的一段代码：

switch (event.getAction()) {  
                case MotionEvent.ACTION_UP:  
                                   //.....
                                                if (!post(mPerformClick)) {  
                                                                                    performClick();  
                                                                                                                }  
                                                                        }  
                }

performClick内的代码如下：

public boolean performClick() {  
        sendAccessibilityEvent(AccessibilityEvent.TYPE_VIEW_CLICKED);  
            if (mOnClickListener != null) {  
                        playSoundEffect(SoundEffectConstants.CLICK);  
                                mOnClickListener.onClick(this);  
                                        return true;  
                                            }  
                return false;  
                    }

所以，本质上，setOnClickListener里面的onClickListener是在onTouchEvent里被响应的

第一条路线：从下向上的事件分发机制

要想要onClickListener被响应，除了要set到View上面之外，还需要能让点击的事件可以被传进来，而不是被拦截掉。

点击的事件，会先触发第一条从下而上（从父View到子View）的路线：事件分发路线

一个点击事件会首先被传到onInterceptTouchEvent方法，这个方法是负责决定事件的传递方向的。onInterceptTouchEvent的返回值是boolean，如果return true，就意为着这个事件被自身消费掉，传导到自身的onTouchEvent而不传导到子控件。
在这个时候需要注意事件发生的顺序，在ACTION_MOVE之前一定会有ACTION_DOWN，如果在ACTION_DOWN的时候已经return，则对ACTION_MOVE里面写了事件处理，也不可能被触发

那么，接下来有两种可能

  1. return true，事件被拦截，使用该View的onTouchEvent方法(没有被setOnTouchListener的情况)或者onTouchListener的onTouch方法(被setOnTouchListener的情况)

    1. return false，事件未被拦截，传导到下面的View中去

    子View的onInterceptTouchEvent也会有响应，决定事件是否向下传递

    继续下去，事件可能就被最终的某个View上

第二条路线：从上向下的事件响应机制

第二条路线的关键是onTouch和onTouchEvent这两个方法：

对于onTouch，我们要想写这个方法，首先需要对View setOnTouchListener，并对onTouchListsner的onTouch方法进行重载
而对于onTouchEvent，我们是直接对View里面的这个方法进行重载的
相比较而言，可以这么说：onTouchEvent是在内改，而onTouch是在外对View一个属性的修改

在调用上，这两个方法都是在View的dispatchTouchEvent中被调用的，但是onTouch会优先于onTouch执行。

而onTouch和onTouchEvent也都是有返回值的，true意味着事件不再向下传导，自己消费，false意味着事件继续向下传导，父View还可以继续响应。

这也就意味着，如果onTouch返回true，那么onTouchEvent将不再被响应。
我们来看一下View的dispatchTouchEvent方法，可以看的更明显。

    public boolean dispatchTouchEvent(MotionEvent event) {  
                if (mInputEventConsistencyVerifier != null) {  
                                mInputEventConsistencyVerifier.onTouchEvent(event, 0);  
                                        }  

                        if (onFilterTouchEventForSecurity(event)) {  
                                        //noinspection SimplifiableIfStatement  
                                        ListenerInfo li = mListenerInfo;  
                                                    if (li != null && li.mOnTouchListener != null && (mViewFlags & ENABLED_MASK) == ENABLED  
                                                                                && li.mOnTouchListener.onTouch(this, event)) {  
                                                                        return true;  
                                                                                    }  

                                                                if (onTouchEvent(event)) {  
                                                                                    return true;  
                                                                                                }  
                                                                        }  

                                if (mInputEventConsistencyVerifier != null) {  
                                                mInputEventConsistencyVerifier.onUnhandledEvent(event, 0);  
                                                        }  
                                        return false;  
                                            }

还有一点需要注意的是，要想响应onTouch，那么这个点击的控件必须是enable的，要是这个控件是非enable的，那我们只能重载onTouchEvent了。

传递到某个子View的时候，此时会响应这个View的onTouch或onTouchEvent方法，根据他的返回值决定是否向下传递，那么接下来也是两种可能：

  1. onTouch/onTouchEvent return true ，本身这个View把这个事件消费了
  2. onTouch/onTouchEvent return false ，响应完后再向父View传递这个事件。

解决事件冲突，核心就是处理onInterceptTouchEvent，onTouch，onTouchEvent这三个方法的返回值。

点击事件处理小技巧：Gesture

实际上在真机使用时，点击不仅会触发到ACTION_DOWN事件，由于手指可能会颤抖一下，还会触发到ACTION_MOVE，我们可以设定一个继承自GestureDetector.SimpleOnGestureListener的类，通过重载里面的onScroll，监听X，Y轴的坐标移动量，返回true/false。
mGestureDetector.onTouchEvent(event)会返回和onScroll同样的返回值，通过这个方法结合onInterecptTouchEvent，可以使点击更顺畅。
